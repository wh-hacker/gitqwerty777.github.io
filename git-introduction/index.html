Git 觀念

  • 分散式開發
    • 每個人都有完整的容器，各自獨立
    • 不需要中央管理

Git 安裝

  • Debian/Ubuntu: 主要: git-core 和 git-doc
    • 選擇性: git-gui, gitk(圖形化), gitweb …
  • Windows: Cygwin, msysGit
    • 或是 github 安裝版

Git 常用名稱

  • origin： 遠端(remote) Repository 的別稱，預設即為 origin/master 分支
  • master： 本地(local) Repository 的 master 分支
    • 名為 master 的分支通常是專案中的主要分支
  • HEAD
    • 目前分支的最新一個提交
  • 相對名稱
    • HEAD/HEAD^/HEAD^^/HEAD~4
      • 代表目前版本 / 前一版 / 前二版 / 前四版的提交
    • HEAD^1/HEAD^2/ …
      • 代表第一個父提交，第二個父提交 ……
    • A…B
      • A 和 B 的相對差異
      • 列出 A 和 B 的祖先，直到 A 和 B 的分歧點
  • git rev-parse HEAD: 取得代稱 (HEAD^, HEAD~1, tag) 的辨識碼
  • ORIG_HEAD：合併或重設前的 HEAD
  • FETCH_HEAD：fetch 抓取的 HEAD
  • MERGE_HEAD：合併時，另一個分支的 HEAD

Git 資料結構

Index 和 Data:[Blobs, Tree, Commits, Tags]

  • Data
    • Blob(binary large object): 檔案本身
      • 新增檔案時，看的是檔案的 sha1，而不是檔名
        • 若有兩個檔案內容相同，則 git 只會有一份 blob
    • Tree: 目錄資訊，指向 Blobs
      • 記錄 Blob 的辨識碼，檔案的資訊(名稱)，子 Tree 等
      • Git 可以用 Tree 快速產生兩個版本間的差異
    • Commit: 更動資訊，新的 Commit 會指向本來的 Commit
      • 包含作者，時間，commit 對應的目錄(Tree)，commit message
    • Tag: 記錄 commit 的別名
      • lightweight tag
        • 容器私有的
      • annotated tag
        • 真實物件
  • Index
    • 即為 Staging area(暫存區)
    • 可視為一顆 Tree，在 add 之後就將新檔案加入 Tree 中
    • 正式 commit 時，index 即為 commit 的 Tree

打包機制(pack file)：找內容相似的檔案，只儲存一份 + 差異的部分

分支

  • 可視為指向某個 commit 的指標
  • 切換的時候工作目錄 (容器的目錄) 會被改變
  • 用途
    • 分隔 測試、開發、穩定的版本
    • debug 分支
    • feature 分支
  • 可以分層
    • “bug/…”
    • “release/…”
  • 開發時用 branch(動態)，完成後可用 tag 取代(靜態)
  • 分支是 local 的
    • 平常上傳只會將目前的 HEAD 和遠端分支的 HEAD 同步

合併

每次提交和合併的時間間隔愈短，每次的衝突愈少，較容易實作。

同一條線上的合併

  • 目前版本比被合併版本新 → Already up to date
  • 目前版本比被合併版本舊 → fast-forward
    • 將目前版本移動到被合併版本的 HEAD
