Git 常用指令

主要參考此書：版本控制：使用 Git

Git 基本觀念

檔案階段 檔案所在工作區 說明
已修改 Working Directory(目前的目錄) 實際進行檔案修改的地方
已暫存 Staging area(暫存區) 下次要提交的變動清單
已提交 Repository(容器) 包含已提交的檔案及變動記錄

其他檔案

  • 被忽略： 在名為 .gitignore 的檔案中修改想忽略的檔案，如暫存檔、log、或編譯後的產物、筆記
    • 每個目錄都可以有自己的 .gitignore 檔，以較接近檔案的目錄為準
    • 可以用 git add 強制將被忽略的檔案加入暫存區
  • 觀看檔案的狀態: git status

提交(commit，送交)

  • 容器變動的單位
  • 不一定所有檔案都要提交
    • 只要不將檔案加入 staging area，就不會被提交
  • 提交時機
    • 處於穩定狀態
      • 測試完成
      • 移動函式至另一個檔案
    • 結束工作前

辨識碼 

- 所有物件都有一個獨特的辨識碼
    - SHA1 hash
    - 包含 commit, tag, 檔案……
- 通常取前幾碼

基本流程: 修改檔案 => 將修改的檔案加入 staging area (git add) => 提交變更 (git commit) => 繼續修改其他檔案
可以在線上容器和本機容器間進行同步

設定

第一次安裝 git 之後，先設定自己的名字與信箱，因為 Git 用名字與信箱來分辨貢獻者

若不加 --global 則設定只會在該容器生效

設定 commit 的 template(預設訊息格式)

Git 常用指令

Git 檔案操作

要從 Git 刪除或移動檔案，需要將變動加入 staging area(暫存區)
(直接刪除 / 移動不會影響版本中的檔案，因為這項變動沒有提交)

Git 版本更動

存取遠端容器

push：如果遠端容器已經更改過，可能需要先 pull，在本地 merge 後再 pull
pull 機制：先 fetch 遠端的 branch，然後與本地端的 branch 做 merge，會多產生一個 merge commit

Git 分支(branch)

Git Tag(標籤)

git tag [tagname] [commit] # 新增 tag 至 commit(可為中文) 
git tag -d tagname # 刪除 tag

Git 合併(merge) 分支

產生衝突時：先用 git diff/status 查看
衝突的地方為如下格式

分別是兩個版本的差異。
將其修改後即可 add & commit ，產生合併提交

git diff --ours/--theirs 可以查看原版本和衝突版本的差異(ours→合併的, theirs→被合併的)

warning: git diff 用在衝突檔案時只會顯示衝突的部分，而不會顯示只有一邊修改的部分

rebase: 重新指定提交的基礎位置

可進行重新排序，編輯，移除，擠壓，拆散提交，是個強大的指令

  • 優點：可以不產生分支線和 merge commit
  • 危險
    • 等於改變提交記錄，僅適合還沒上傳的 local branch
    • commit 被移動，所以特徵碼會改變 (也就是 原本的 commit 會消失)
    • commit 的形狀會被改變(一條直線)
      • 可用 --preserve-merge 參數保留形狀
  • 麻煩
    • 移動的 commit 若屬於多個分支，則每個分支都要重新指定位置

使用時機：將自己開發的 branch rebase 到 master

修改一個分支的歷史—使用 rebase
git rebase -i [previous commit in the same branch]

merge還是 rebase?
需要保留樹狀記錄就用 merge，反之用 rebase

amend: 修改上一次的 commit

  • 取出 Staging Area 的資料並 合併到最後一次的提交
  • 若在最後一次提交後沒有做過任何修改，那麼唯一有更動的是提交時的訊息
  • 使用時機
    • commit message 的修改
    • 忘記 add/ 多 add 檔案 的時候
    • 發現一些 小錯誤

reset: 移動至指定版本

將 HEAD 視為指標，平常是指向最新的一次提交。但 git reset 可以移動 HEAD 到指定的提交版本
(附註：checkout, commit 等指令都會改變 HEAD 指標的位置)

git reset [commit] [--hard/--soft/--mixed] #  還原到指定提交的狀態
--soft 只移動 HEAD 指針，其他不變    
--mixed 暫存區也被改變，但工作目錄不變 (預設值) 
--hard 暫存區和工作目錄都被改變 
git reset HEAD filename # 將檔案從 staging area 移除 (= 將暫存區的這個檔案還原到 HEAD 時的狀態)

checkout

使檔案還原到指定提交時的狀態

使用時機：
發現檔案改錯，用 git checkout (HEAD) filename 把檔案還原到最近的版本

git checkout [HEAD/commit 前四碼 /--] filename # 檔案會被還原到指定的 commit 時的檔案

revert: 新增「還原」的提交

新增一筆 commit，做相反的變更。等同於還原至指定版本
使用時機: 你的送交已經上傳，別人有相同的一份記錄，所以你無法用 git reset, git commit --amend 修改歷史記錄

git log 可以用來查詢提交的歷史記錄及其細節，有很多參數可以用，建議用 GUI(gitk, git gui) 比較清楚

尋找提交: bisect, blame

找尋發生問題的提交

用檔案尋找提交

Git stash

把目前工作區 (working directory) 的東西丟到暫存區 (stash) 裡，等之後在回來拿。

使用時機

  1. 你不得不修改一個緊急 Bug，你可以先把目前工作狀況丟到 stash，這時候你的工作區和上次剛提交內容的狀況一樣，等到修完 Bug 後再把 stash 中剛剛做到一半的東西叫回來繼續
  2. 今天的工作時間結束後，還有未完成的部分

  3. pull 到一個不乾淨的樹而不用 merge

  4. 保留自己的設定(如 makefile)
